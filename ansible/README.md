# DeployForge — Stage 3 of 4

**Ansible: kubeadm cluster bootstrap + Calico CNI + monitoring stack + app deployment**

This stage takes the 3 bare Ubuntu EC2 instances Terraform created and turns them into a working production-grade Kubernetes cluster, then deploys the Spring Boot app.

## What this stage does

```
For each of the 3 EC2 instances (master + 2 workers):
  1. Common base: kernel modules, sysctl, swap off, hostname
  2. Install containerd 1.7 with SystemdCgroup enabled
  3. Install kubeadm/kubelet/kubectl 1.29 with apt-mark hold

On the master only:
  4. kubeadm init (bring up control plane)
  5. Set up admin kubeconfig for ubuntu user
  6. Apply Calico v3.27 CNI manifest
  7. Install Helm
  8. Install local-path-provisioner (default StorageClass for PVCs)
  9. Install ingress-nginx via Helm (NodePort 30080/30443)
 10. Install kube-prometheus-stack (Prometheus + Grafana + AlertManager)

On each worker:
 11. kubeadm join using a fresh token from the master

Then deploy-app.yml separately:
 12. Copy ../k8s/base manifests to master
 13. Render real Secret with DB credentials
 14. kubectl apply -k (Kustomize base)
 15. Wait for postgres + taskmanager rollouts
```

## Prerequisites

- Stage 2 (Terraform) applied — 3 EC2s running
- `make inventory` was run in `terraform/`, producing `ansible/inventory.ini`
- SSH from your Mac to all 3 instances works (you verified this)

## How to deploy

```bash
cd ansible/

# 1. Install required Ansible collections (one time)
make collections

# 2. Verify SSH to all nodes via Ansible
make ping
# Should show all 3 hosts with green "SUCCESS"

# 3. Run the full bootstrap (~15-20 minutes)
make bootstrap
# Watches each task. If anything fails, the playbook stops with a clear error.

# 4. Get a kubeconfig you can use locally
make kubeconfig
export KUBECONFIG=~/.kube/deployforge.config
kubectl get nodes
# Should show:
#   NAME                    STATUS   ROLES           AGE   VERSION
#   deployforge-master      Ready    control-plane   5m    v1.29.4
#   deployforge-worker-1    Ready    <none>          3m    v1.29.4
#   deployforge-worker-2    Ready    <none>          3m    v1.29.4

# 5. Deploy the Spring Boot app
DB_PASSWORD=secretpass123 make deploy-app

# 6. Verify it works
make smoke
```

## Key design decisions for the report

| Decision | Why |
|---|---|
| **Roles, not flat playbooks** | Reusable, testable per-component. Industry standard structure. |
| **containerd, not Docker** | Docker shim was removed from Kubernetes 1.24. containerd is what the official kubeadm docs recommend. |
| **`SystemdCgroup = true`** | kubelet on Ubuntu 22.04 uses systemd as its cgroup driver. Mismatched runtime → kubelet refuses to schedule pods. This is the #1 silent failure mode for kubeadm clusters. |
| **`apt-mark hold` on kube* packages** | An accidental `apt upgrade` would skew kubelet vs kubeadm versions and break the cluster. |
| **Calico, not Flannel** | Calico enforces NetworkPolicy (Flannel doesn't). Our k8s/base/networkpolicy.yaml needs a CNI that enforces it. |
| **local-path-provisioner as default StorageClass** | kubeadm doesn't ship a dynamic provisioner. Without one, the postgres StatefulSet's PVC stays Pending forever. |
| **kube-prometheus-stack via Helm** | Prometheus + Grafana + node-exporter + AlertManager + ServiceMonitors all configured together. Doing this by hand is hours of YAML. |
| **NGINX ingress on NodePort** | LoadBalancer requires AWS cloud-controller-manager (which we'd need to install separately for self-managed clusters). NodePort works out of the box. |
| **`controlPlaneEndpoint` in kubeadm config** | Makes joining workers easier and lets you swap the master without regenerating certs. |
| **Idempotent everywhere** | Re-running `make bootstrap` after a partial failure picks up where it left off. |

## How to access things after bootstrap

| What | How |
|---|---|
| **kubectl from your Mac** | `make kubeconfig && export KUBECONFIG=~/.kube/deployforge.config` |
| **Spring Boot app** | `kubectl port-forward -n deployforge svc/taskmanager 8080:80` then `curl http://localhost:8080/api/tasks` |
| **Grafana** | `http://<master-public-ip>:30030` — login admin / admin |
| **Prometheus** | `http://<master-public-ip>:30090` |
| **NGINX ingress** | `http://<master-public-ip>:30080` (host header `taskmanager.local` per Stage 1 ingress) |

## Common errors and fixes

| Symptom | Likely cause | Fix |
|---|---|---|
| `kubeadm init` hangs at "Initial timeout of 40s passed" | swap is on, or `SystemdCgroup=false` | Re-run `roles/common` and `roles/containerd` — their tasks fix this |
| Worker stays NotReady | Calico pods CrashLoopBackOff | `kubectl logs -n kube-system <calico-pod>` — usually IP allocation conflict. Make sure pod CIDR `192.168.0.0/16` doesn't conflict with VPC CIDR `10.0.0.0/16` (they don't, but double-check). |
| `coredns` pods Pending forever | No CNI installed | site.yml step 4 installs Calico. If you skipped to step 5 by mistake, re-run from start. |
| `make ping` fails with timeout | Security group blocks Ansible's SSH | Confirm `aws_security_group.ssh` has your current IP. If your IP changed: update `terraform.tfvars`, run `terraform apply`, then `make inventory` again. |
| `helm: command not found` during master role | Internet egress blocked | Check the master's outbound SG rule (allow all should be set) |
| `kubectl get nodes` shows version mismatch | Mixed version pkgs | Re-run `roles/kubernetes` — it pins all 3 binaries to `k8s_full_version` |

## Re-running after teardown/rebuild

If you `terraform destroy` + `terraform apply` and get fresh EC2s with new IPs:

```bash
cd ../terraform && make inventory     # rewrites ansible/inventory.ini with new IPs
cd ../ansible && make bootstrap        # cluster comes up fresh in ~15 min
make kubeconfig                        # new kubeconfig with new master IP
make deploy-app
```

## What this stage does NOT do

- **Doesn't push your Docker image to ECR.** You either build/push manually or wait for Stage 4 (CI/CD) to do it. For first deploy, the deployment.yaml's image tag is `ghcr.io/deployforge/taskmanager:latest` which won't exist in your ECR — you'll see ImagePullBackOff. That's expected; Stage 4 fixes it.
- **Doesn't set up a real domain.** The Ingress uses `taskmanager.local`. Add to `/etc/hosts` or use port-forward.
- **Doesn't enable HTTPS.** For a course project, HTTP via NodePort is fine. Production would add cert-manager + Let's Encrypt.
- **Doesn't back up etcd.** A real cluster runs etcd snapshots to S3. For a demo lasting hours, not needed.
